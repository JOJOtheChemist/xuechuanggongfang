# 高考学校搜索工具参数设计说明

## 这份文档只讨论一件事
不是讨论 skill。

不是讨论“tool 内部怎么帮大模型兜底清洗自然语言”。

这份文档只讨论：

- 搜索学校这套工具，应该怎么设计参数
- 大模型面对自然语言提问时，应该决定传什么参数
- 哪些信息该传给 `query`
- 哪些信息必须拆成独立字段
- 理科、文科、冲稳保、学校详情，这几类问题分别该怎么调工具

## 核心判断
你说得对，关键点不该是“tool 再帮忙洗一遍句子”。

关键点应该是：

1. 工具参数定义要清晰。
2. 大模型要能看懂这些参数分别代表什么。
3. 大模型在调用 tool 时，要主动把自然语言拆成结构化参数。
4. tool 最多做轻量兜底，不能把“正确拆参”这件事本身扛过去。

也就是说，真正要优化的是：

- `tool schema`
- `tool prompt`
- `agent 的 tool 使用规则`

而不是把自然语言一股脑丢给 `query`，然后指望 tool 自己猜。

## 现在这套学校搜索相关工具
当前主要有 4 个：

- `search_yunnan_admission`
- `list_yunnan_admission_schools`
- `search_yunnan_admission_school_detail`
- `interpret_admission_risk`

从职责上讲，应该这样理解：

### 1. `search_yunnan_admission`
用途：

- 综合检索入口
- 适合“按分数/科类/专业方向找学校”
- 适合“推荐一批学校”

它应该吃的是“结构化筛选条件”，不是整句问法。

### 2. `list_yunnan_admission_schools`
用途：

- 更偏列表罗列
- 适合“按城市/学校层次/公办民办/冲稳保分档”筛学校

它也应该主要吃结构化条件。

### 3. `search_yunnan_admission_school_detail`
用途：

- 用户已经明确问某一所学校
- 比如“昆明医科大学怎么样”“云南大学新闻学值不值得报”

这个工具的核心入参应该是：

- `institution_id`
- 或者非常明确的 `institution_name`

不应该把“学校名 + 专业名 + 情绪词 + 问句”整个塞进去。

### 4. `interpret_admission_risk`
用途：

- 解释“冲、稳、保”
- 把风险档位翻译成家长和学生听得懂的话

这个工具不负责找学校，只负责解释。

## 参数设计原则

## 原则 1：`query` 不是垃圾桶
`query` 不应该变成“用户原话全文塞进去”的兜底字段。

`query` 应该只承担一种职责：

- 补充学校名、专业词、城市词这类简短检索关键词

比如：

- `云南大学`
- `新闻学`
- `昆明`
- `金融`

而不是：

- `云南文科450分，喜欢新闻，推荐什么学校`
- `用张雪峰视角看下金融值不值得报`

这些内容里：

- 分数应该进 `score`
- 科类应该进 `subjectTrack`
- 专业方向应该进 `majorKeyword`
- 风险意图应该进 `riskBucket`
- “值不值得报”根本不是 query，它是回答方式

## 原则 2：分数、科类、专业、城市，必须拆字段
下面这些信息，不应该混在 `query` 里：

- `score`
- `rank`
- `examType`
- `subjectTrack`
- `majorKeyword`
- `majorCategory`
- `city`
- `schoolLevel`
- `ownershipType`
- `riskBucket`

原因很简单：

- 这些是稳定筛选条件
- 大模型能识别出来
- tool 也最容易直接消费

如果这些信息还混在一句自然语言里，就等于大模型没有真正完成“理解用户问题”。

## 原则 3：学校名明确时，不要再走模糊列表思路
如果用户已经明确问的是：

- `昆明医科大学口腔医学怎么样`
- `云南大学新闻学值不值得报`

那就应该优先走 `search_yunnan_admission_school_detail`。

这时最重要的是：

- `institution_name=昆明医科大学`
- `major_category` 或 `majorKeyword=口腔医学`

而不是：

- `query=昆明医科大学口腔医学怎么样`

## 原则 4：冲稳保是意图，不是 query
如果用户说：

- `帮我按冲稳保看看学校`

重点不是提取出某个关键词，而是要识别出用户意图：

- 他要按风险分层看学校

所以应该重点传：

- `score`
- `subjectTrack`
- `riskBucket`

甚至可以多次调用：

- 一次 `hard`
- 一次 `stable`
- 一次 `safe`

而不是：

- 把“冲稳保看看学校”塞进 `query`

## 建议的大模型拆参规范

## A. 理科分数找学校
用户：

```text
云南理科560分，帮我按冲稳保看看学校
```

大模型应该理解成：

- 省份：云南
- 分数：560
- 科类：理科，对应 `subjectTrack=物理类`
- 意图：冲稳保分档
- 未明确学校名
- 未明确专业词

推荐调用方式：

### 调法 1：分三次查

```json
{
  "score": 560,
  "examType": "gaokao",
  "subjectTrack": "物理类",
  "riskBucket": "hard",
  "limit": 5
}
```

```json
{
  "score": 560,
  "examType": "gaokao",
  "subjectTrack": "物理类",
  "riskBucket": "stable",
  "limit": 5
}
```

```json
{
  "score": 560,
  "examType": "gaokao",
  "subjectTrack": "物理类",
  "riskBucket": "safe",
  "limit": 5
}
```

这比一次只传：

```json
{
  "query": "云南理科560分，帮我按冲稳保看看学校"
}
```

要清楚太多。

## B. 文科分数 + 专业兴趣
用户：

```text
云南文科450分，喜欢新闻，推荐什么学校
```

大模型应该理解成：

- 分数：450
- 文科，对应 `subjectTrack=历史类`
- 专业兴趣：新闻
- 文科问题里，除了专业，还应该更重视：
- 学校地理位置
- 学校层次
- 院校氛围

推荐调用参数：

```json
{
  "score": 450,
  "examType": "gaokao",
  "subjectTrack": "历史类",
  "majorKeyword": "新闻",
  "limit": 5
}
```

如果用户还说了城市偏好，比如“想去昆明或者成都”，那就再补：

```json
{
  "city": "昆明"
}
```

或者如果工具支持多城市，再补 `preferredCities`。

这里不建议把：

- `喜欢新闻`
- `推荐什么学校`

塞进 `query`。

## C. 学校详情
用户：

```text
昆明医科大学口腔医学怎么样
```

大模型应该理解成：

- 用户不是在找一批学校
- 用户是在问一所明确学校
- 学校名：昆明医科大学
- 专业方向：口腔医学

推荐调用参数：

```json
{
  "institution_name": "昆明医科大学",
  "major_category": "口腔医学",
  "exam_type": "gaokao"
}
```

不建议这样传：

```json
{
  "institution_name": "昆明医科大学口腔医学怎么样"
}
```

因为这样会把“学校名”和“问题尾巴”混在一起，容易误命中。

## D. 学校 + 专业值不值得报
用户：

```text
云南大学新闻学值不值得报
```

大模型应该理解成：

- 学校名：云南大学
- 专业方向：新闻学
- 用户想要的不是“搜学校”，而是“先查详情，再解释值不值得”

推荐链路：

1. 先调用 `search_yunnan_admission_school_detail`

```json
{
  "institution_name": "云南大学",
  "major_category": "新闻学",
  "exam_type": "gaokao",
  "subject_track": "历史类"
}
```

2. 再基于返回结果，决定要不要补 `interpret_admission_risk`

## E. 理科更看专业，文科更看学校与城市
这条应该直接写进 Agent 的用 tool 规则里。

### 理科问题
优先拆：

- `score`
- `subjectTrack`
- `majorKeyword`
- `majorCategory`

因为理科很多时候用户更在意：

- 专业是不是强
- 专业以后就业方向
- 学校是不是能托住这个专业

### 文科问题
优先拆：

- `score`
- `subjectTrack`
- `city`
- `schoolLevel`
- `majorKeyword`

因为文科很多时候用户更在意：

- 学校在哪
- 层次怎么样
- 城市资源和实习资源
- 专业名字听起来好不好，不一定比城市更重要

## 当前工具最需要优化的不是“清洗”，而是这几件事

## 1. tool prompt 要更直接
现在 tool prompt 虽然已经比以前清楚，但还可以更明确写死：

- 不要传整句自然语言
- 分数单独传 `score`
- 科类单独传 `subjectTrack`
- 专业方向单独传 `majorKeyword`
- 学校详情问题优先传 `institution_name`
- 冲稳保问题优先按 `riskBucket` 分档查询

## 2. agent prompt 要更明确区分“学校列表”和“学校详情”
现在最大的问题之一，不是 tool 本身不能查，而是：

- 用户明明是在问某一所学校
- 大模型却可能还按“学校搜索列表”思路去调

所以 Agent 层必须更明确：

- 问一批学校，用 `search_yunnan_admission` / `list_yunnan_admission_schools`
- 问一所学校，用 `search_yunnan_admission_school_detail`

## 3. 文科和理科的拆参策略要分开写
这个不能只靠回答时体现，应该在工具调用规则里就体现：

- 理科优先专业导向
- 文科优先学校层次 + 城市导向

## 4. `query` 应该降级为“补充字段”
我建议后面把 `query` 的定位写成：

- 可选补充
- 只放短关键词
- 没必要时可以为空

不要把 `query` 当主入口。

真正的主入口应该是这些结构化字段：

- `score`
- `rank`
- `subjectTrack`
- `majorKeyword`
- `city`
- `riskBucket`
- `institution_name`

## 建议后续直接补到代码里的规则
如果后面你要改代码，我建议优先改这三层：

### 1. `search_yunnan_admission/prompt.ts`
明确写：

- `query` 只允许短关键词
- 不能把整句问法原样传进来
- 文科和理科拆参规则不同

### 2. `search_yunnan_admission_school_detail/prompt.ts`
明确写：

- `institution_name` 只放学校名
- 专业方向单独进 `major_category`
- 不要把“怎么样”“值不值得报”这类问句尾巴传进去

### 3. `yunnan-gaokao-volunteer-consultant-v3.ts`
明确写：

- 文科优先学校、城市、层次
- 理科优先专业、分数匹配
- 冲稳保问题优先分档多次查询
- 学校详情问题优先 detail tool

## 一句话总结
学校搜索工具真正该优化的重点，不是“tool 内部继续帮大模型洗自然语言”，而是让大模型在调用前就把自然语言拆成清晰参数：

- 分数归分数
- 科类归科类
- 专业归专业
- 学校名归学校名
- 冲稳保归风险档位

只有这样，tool 返回结果才会稳定，Agent 后面的解释才有意义。
