# 学校卡片一直渲染失败调试关键笔记

日期：2026-04-14

## 现象

志愿页顶部筛选胶囊能显示“64 所学校 / 当前筛选 64 所”，说明接口数据和统计数量已经到了前端，但下面具体学校卡片一直不显示。

这个问题不要只盯着接口数量看：数量能显示，只代表列表数据参与了统计，不代表小程序渲染层一定能稳定渲染卡片列表。

## 最关键结论

这次最终稳定显示，靠的是两件事一起做：

1. 后端给每个学校补出数据库级稳定 ID，并在 API 里返回 `stableId`。
2. 前端不要再走不稳定的子组件列表渲染路径，而是像左上角下拉组件一样，先在页面层生成稳定的展示数组，再直接在页面模板里 `v-for` 渲染。

最终前端使用的是：

- `visibleSchoolCards`：页面层已经整理好的学校卡片数组。
- `school.renderKey`：卡片渲染专用稳定 key。
- `school.tags[].renderKey`、`school.majors[].renderKey`：标签和专业渲染专用稳定 key。

也就是说，模板里不要现场拼复杂 key，也不要依赖不稳定对象引用；先在 JS 里把渲染字段整理好，再交给模板。

```vue

```

## 为什么要学左上角下拉组件

左上角下拉能显示，是因为它的循环列表不是在模板里临时凑数据，而是先把数据整理成可渲染的选项数组，然后模板只负责简单地循环。

这次学校卡片也改成这个思路：

- 数据整理放在 computed / JS 映射里。
- 模板只拿稳定数组渲染。
- 每一层循环都用明确的 `renderKey`。
- 不把复杂对象、临时拼接、可能缺失的字段直接塞进模板 key。

这个方案对小程序渲染层更稳。

## 以后再遇到类似问题的排查顺序

1. 先看 API 是否真的返回学校数组，不只看总数。
2. 再看每个学校是否有稳定 ID，例如 `stableId`。
3. 如果顶部数量有、卡片没有，优先怀疑渲染路径，不要马上怀疑接口没数据。
4. 学校卡片这类大列表，优先在页面层生成 `visibleSchoolCards`，不要让模板现场加工复杂结构。
5. `v-for` 的 `:key` 优先用预先生成好的 `renderKey`，不要用临时表达式乱拼。
6. 如果子组件渲染不稳定，先改成页面内联渲染验证，就像这次模仿左上角下拉组件一样。
7. 后端如果新增稳定字段，上线必须执行迁移、构建、PM2 重启，再做 API 冒烟测试。
8. HBuilder / 小程序端如果仍看不到，清缓存并重新编译；这类问题有时会被旧包缓存误导。

## HBuilder 编译注意

这不是这次学校卡片的根因，而是老编译工具和新 Node 版本不兼容。后续如果要命令行编译，优先使用 HBuilderX 自带的 Node，或者直接用 HBuilderX GUI 编译运行。

## 一句话复盘

这次不是“接口完全没数据”，而是“数据到了、统计能用，但卡片列表渲染路径不稳”。后端补 `stableId`，前端页面层生成 `renderKey` 并内联渲染后，学校卡片终于稳定显示。
