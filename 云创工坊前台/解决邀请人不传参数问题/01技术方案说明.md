# 解决邀请人参数丢失的技术方案：【桌号模式】 🛡️

## 1. 核心问题定位

**原有的【柜子模式】（数据缓存）存在以下风险：**
- **不稳定性**：用户扫码后，如果小程序发生冷启动、内存清理或者在登录页强制刷新，保存在 `uni.setStorageSync` 里的数据可能失效或无法及时读取。
- **传递断层**：从 `扫码 -> 启动页 -> 登录页 -> 业务/团队页`，链路太长，缓存容易在中间环节“丢包”。

---

## 2. 解决方案：【桌号模式】（URL 路由传参）

就像去饭店吃饭，不再由服务员把桌号记在柜台（缓存），而是**客人在扫码的一瞬间就把桌号牌（邀请人 ID）挂在身上**。

### 🚀 技术实现逻辑：

1.  **扫码识别 (App.vue)**：
    小程序启动时解析 `scene`。如果不带登录态，直接将 `inviter_id` 拼接到登录页 URL 中。
    - **URL 示范**：`/pages/auth/login/index?inviter_id=6923...&type=business`

2.  **登录承接 (login/index.vue)**：
    登录页不需要去翻“柜子”，直接从 `onLoad(options)` 中提取 `inviter_id`。这样无论页面如何重启，只要地址栏没变，桌号（邀请人）就永远在。

3.  **数据固化 (Cloud API)**：
    在点击“微信一键登录”的一瞬间，将 `inviter_id` 发送给 `user-center` 云函数。云端在创建用户账号时，同步完成绑定。

4.  **精准重定向**：
    登录成功后，根据 URL 里的 `type`，自动带着参数跳转到最终的业务页（报名表或团队确认页）。

---

## 3. 需要上传/部署的云函数

执行完代码修改后，请务必**右键上传/部署**以下云对象：

1.  **`user-center`**：
    - 关键更新：`loginByWeixin` 方法。
    - 作用：接收前端透传的“桌号”，实现新用户注册时的即时绑定。

2.  **`team-service`**：
    - 关键更新：添加了详细的服务端流水日志。
    - 作用：记录团队加入过程，防止邀请逻辑在服务端静默失败。

---

## 4. 如何验证已经解决？

在微信开发者工具或真机测试时，观察以下日志标识：

- **第一步 (App)**：`[App] 未登录，携带“桌号”跳转登录页`
- **第二步 (Login)**：`[login] 从 URL 获取到桌号 - inviterId: 6923...`
- **第三步 (Cloud)**：查看云函数运行日志，应显示 `--- [user-center][loginByWeixin] 入口 ---` 以及正确接收到的 `inviterId`。

---

> **总结**：只要登录页面能显示出邀请人 UID，就说明“桌号”已经成功传递。即使用户退出了重开，只要扫码进入，这个关系就绝对不会丢失。
