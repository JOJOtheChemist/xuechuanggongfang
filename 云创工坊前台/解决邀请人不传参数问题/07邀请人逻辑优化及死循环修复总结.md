# 07 邀请人逻辑优化及死循环修复总结

针对“邀请人丢失”及“iOS 缓存不可靠”问题的全案解决总结。

## 1. 核心逻辑优化：从“依赖缓存”到“参数硬传”
由于 `uni.setStorageSync` 在 iOS 或长时间切后台时可能失效，我们将逻辑改为 **URL 优先**：
- **链式传递**：从扫码进入中转页（`invite-handler`），到登录页（`login`），再到最终的报名/确认页，邀请人 ID 始终通过 URL 参数 `inviter_id=...` 显式携带。
- **缓存兜底**：`storage` 仅作为临时的二次备份，不再是唯一的判定标准。

## 2. 后端“懒绑定” (Lazy Binding)
解决了登录用户扫码不记录的问题：
- **即时补录**：在 `user-center` 增加了 `bindInviter` 接口。用户扫码点击“立即查看”的瞬间，前端即发起补录请求。
- **业务锁死**：在“提交报名”或“支付入团”的后端逻辑中，最后一次检查并补录终身推荐人，确保只要建立了业务关系，邀请人奖励和拉新日志（`invite_logs`）绝不漏统计。

## 3. 彻底修复“导航死循环”与超时报错
扫码进入时出现的 `reLaunch:fail timeout` 报错已完美修复，修复方案如下：

### 故障原因分析：
- **触发重叠**：微信启动时 `onLaunch` 和 `onShow` 几乎同时触发，导致发起两次 `reLaunch`。
- **隐藏循环**：`reLaunch` 导致 App 隐藏再打开，触发 `onHide` -> `onShow`。如果 `onHide` 里错误地重置了状态，会导致无限触发跳转，由于间隔太短（<200ms），微信控制台报超时错误。

### 修复方案：
- **场景锁 (Scene Lock)**：在 `App.vue` 引入 `lastHandledScene`。同一扫码场景在一次运行周期内只允许发起一次跳转。
- **导航优化**：中转页改用 `redirectTo`（替换模式）而非 `reLaunch`（重启模式），大大减轻了微信底层的导航栈负担。
- **沉淀延时**：在中转页跳转至登录页前，增加了 150ms 的“沉淀延时”，确保上一个导航指令已在底层完成渲染。

## 4. 总结
经过此次优化，系统实现了 **“扫码即记、提交即锁、全程透传”** 的拉新机制。即便是在极端的不稳定网络或 iOS 严苛的休眠机制下，也能确保拉新数据的可靠性。
